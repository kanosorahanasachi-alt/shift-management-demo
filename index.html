現状と改善後のイメージ

毎月発生する確認・転記・判断の負担を減らす

この試作版の目的は、シフト作成を人の判断から切り離すことではなく、確認漏れや転記負担を減らし、最終判断をしやすくすることです。

現状：紙・目視・経験依存

  • 紙で希望を提出するため、本人が提出内容を忘れやすい。
  • 提出済み・未提出の確認に手間がかかる。
  • 希望休・有休・勤務不可を目視で確認する必要がある。
  • AM・金銭キー・必要人数などを同時に確認する負担が大きい。
  • 入替・棚卸・研修・事業所訪問などの日別事情が属人的になりやすい。

改善後：入力・一覧・自動チェック

  • スマホまたはPCから希望を入力できる。
  • 提出済み・未提出を一覧で確認できる。
  • 希望休・有休・勤務不可を仮シフト作成時に除外できる。
  • 役割者や必要人数の不足を警告として見える化できる。
  • 日別イベントを登録し、必要人数や勤務区分に反映しやすくなる。

条件整理

必須条件と優先条件を分ける

自動生成ロジックを実務に近づけるには、「絶対に破ってはいけない条件」と「できれば守りたい条件」を分ける必要があります。

必須条件

  • 希望休・有休・勤務不可に勤務を入れない。
  • 連勤上限を超えない。
  • 勤務間インターバルを下回らない。
  • 入替勤務の翌日は休みにする。
  • 必要なAM・金銭キー要員を配置する。
  • 必要人数0の日・勤務区分には配置しない。

優先条件

  • できれば3日程度の勤務ブロックにする。
  • できれば同じ勤務帯を連続させる。
  • 本人希望に近づける。
  • 中番を使って薄い時間帯を補う。
  • 特定スタッフに勤務が偏りすぎないようにする。
  • 着任直後のSMでも判断しやすい形にする。

日別イベント

通常日と異なる条件を反映する

通常日と違う業務がある日は、必要人数や勤務時間帯を補正する必要があります。イベント登録によって、経験則を仕組みに落とし込みやすくします。

イベント 想定される影響 シフト上の補正例
入替 夜間・日跨ぎ勤務が発生する。 入替番を追加し、翌日は自動休みにする。
棚卸 通常業務とは別に確認・作業時間が発生する。 早番または中番の必要人数を補正する。
研修 対象者が一時的に現場から抜ける。 研修時間帯の不足を警告し、中番や代替配置を検討する。
事業所訪問 通常クルーが店外業務で一時的に抜ける可能性がある。 中番配置、または該当時間帯の必要人数増で補う。

実務フロー

実務で使う場合の流れ

いきなり全自動システムにするのではなく、現在の運用を大きく変えずに、確認作業と仮シフト作成を補助する形が現実的です。

希望入力従業員がスマホ・PCから希望を入力。
提出確認管理者が提出済み・未提出を確認。
仮生成必須条件を守って仮シフトを作成。
警告確認不足・違反・修正候補を一覧化。
人が修正SM・作成者が現場判断で調整。
出力掲示・印刷・既存Excelへ反映。

期待できる効果

小さな負担を継続的に減らす

大きな設備投資や大幅な人員削減ではなく、毎月繰り返される小さな負担を減らすことを狙います。

提出確認紙の確認・転記・未提出確認の負担を軽減。
条件確認希望休・有休・勤務不可・役割者不足を見える化。
品質平準化担当者ごとの判断のばらつきを減らす。
全店展開時仮に1店舗あたり月1時間でも削減できれば、全体では大きな改善余地。

実務化する場合

会社管理環境での検証が必要です

このデモは個人作成の試作版です。正式導入する場合は、会社管理環境での検証・権限管理・個人情報管理が必要です。

デモ版の前提

  • すべてサンプルデータで構成。
  • 実際の従業員情報・勤務希望は含まない。
  • GitHub Pages上の静的Webページ。
  • 個人作成の試作であり、正式運用環境ではない。
  • 実運用データを入力しない。

正式運用で必要な確認

  • 会社管理のクラウド・社内システムで運用する。
  • 誰がデータを閲覧・編集できるかを決める。
  • 保存場所・保存期間・削除ルールを決める。
  • 退職者・異動者の権限管理を行う。
  • 有休申請や勤怠ルールとの整合性を確認する。
重要：スマホ希望収集を実運用する場合、個人のGoogleアカウントや個人管理の仕組みではなく、会社が管理する環境で扱う必要があります。この試作版は、あくまで操作感と考え方を確認するためのデモです。
